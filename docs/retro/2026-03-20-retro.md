## Failure Patterns

### Pattern 1: AskUserQuestion in Pipeline Mode

- **Occurrences:** 3 iterations (iter 3, 4, 5)
- **Root cause:** Deploy skill hit crates.io name conflict (`openai-rust` taken), used `AskUserQuestion` to ask user for alternative name. In pipeline mode (`--print`), there is no interactive user to respond. The skill retried the same question across 3 consecutive iterations.
- **Wasted:** 3 iterations (~5 min total)
- **Resolution:** Iter 6 — agent pivoted to GitHub-only deploy after receiving user message via pipeline control file.
- **Fix:** Deploy skill should detect Rust library projects and default to GitHub release. Skip crates.io publish unless `[publish]` is explicitly configured in Cargo.toml with a verified name. Never use `AskUserQuestion` for blocking decisions in pipeline mode — make a reasonable default choice and note it.

### Pattern 2: Circuit Breaker Did Not Fire

- **Occurrences:** 1 (should have triggered after iter 5)
- **Root cause:** `CIRCUIT_BREAKER_LIMIT=3` in `solo-dev.sh` uses hash-based comparison of iter log output. Each deploy iteration had slightly different output (different AskUserQuestion payloads, different tool calls), so hashes never matched as "identical". The breaker checks for identical consecutive failures, but the real pattern was 3 consecutive `continuing` results on the same stage — which should be enough to trip.
- **Wasted:** 0 additional (breaker missed, but iter 6 succeeded anyway)
- **Fix:** Circuit breaker should also count consecutive `continuing` results per stage, regardless of output hash. Add a secondary check: `if same_stage_continuing_count >= CIRCUIT_BREAKER_LIMIT then abort`.
